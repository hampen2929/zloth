# dursor ビジネス戦略

本ドキュメントでは、dursor プロジェクトの方向性とビジネス戦略について検討します。

## 現状分析

| 項目 | 状態 |
|------|------|
| バージョン | v0.1.0 (MVP) |
| 機能成熟度 | 基本機能完成、一部ロードマップ未実装 |
| 技術スタック | FastAPI + Next.js + SQLite（モダン構成） |
| 差別化要素 | マルチモデル並列実行、BYO APIキー、セルフホスト可能 |
| ライセンス | MIT License |

## 選択肢の比較

### 選択肢1: OSSとして公開

#### メリット

| 項目 | 詳細 |
|------|------|
| コミュニティ貢献 | バグ修正、機能追加、ドキュメント改善が期待できる |
| ブランディング | 開発者としての知名度・信頼性向上 |
| 採用・キャリア | ポートフォリオとしてアピール可能 |
| フィードバック | 多様なユースケースからの改善提案 |
| 低コスト運用 | ホスティング・サポートコストを分散できる |
| 競合優位性 | 類似ツール（Cursor, Copilot等）が有料な中、無料で提供できる |

#### デメリット・リスク

| 項目 | 詳細 |
|------|------|
| メンテナンス負担 | Issue対応、PR レビュー、バージョン管理が必要 |
| セキュリティ責任 | APIキー管理など機密性の高い機能があるため慎重な対応が必要 |
| 競合にフォークされる | 商用化されるリスク（MITライセンスなので制限なし） |
| サポート期待 | ユーザーから無料サポートを期待される |
| 収益化困難 | 一度公開すると商用化へのピボットが難しい |

#### 成功の条件

- しっかりした CONTRIBUTING.md と Code of Conduct
- CI パイプライン（テスト、リント）の整備
- 定期的なリリースサイクル
- Discord/Slack などのコミュニティ構築

### 選択肢2: Private として自己利用

#### メリット

| 項目 | 詳細 |
|------|------|
| 自由度最大 | 好きなタイミングで好きな機能を開発できる |
| メンテ負担なし | 他者への対応が不要 |
| 実験的機能 | 破壊的変更も自由に行える |
| セキュリティ | コードが非公開なので脆弱性が悪用されにくい |
| 将来の選択肢 | いつでもOSS化・商用化にピボット可能 |

#### デメリット・リスク

| 項目 | 詳細 |
|------|------|
| 機会損失 | コミュニティからの貢献を受けられない |
| モチベーション | フィードバックがないと開発意欲が低下しやすい |
| 孤立リスク | 車輪の再発明、既知のバグに気づかない可能性 |
| アピール不可 | ポートフォリオとして活用しにくい |
| コスト負担 | 全ての開発・運用コストを自分で負担 |

#### 向いているケース

- 本業の開発を効率化するツールとして使いたい
- まだ機能が不完全で公開に自信がない
- 将来的な選択肢を残しておきたい

### 選択肢3: 商用化

#### ビジネスモデル候補

| モデル | 説明 | 適合度 |
|--------|------|--------|
| SaaS | クラウドでホスティングし月額課金 | ★★★ |
| オープンコア | コア機能はOSS、エンタープライズ機能は有料 | ★★★★ |
| セルフホスト+サポート | セルフホスト版は無料、商用サポート有料 | ★★ |
| マーケットプレイス | Cursor/VSCode拡張として販売 | ★★ |

#### メリット

| 項目 | 詳細 |
|------|------|
| 収益化 | 継続的な収入源になる可能性 |
| フルタイム開発 | 成功すれば専念できる |
| 投資機会 | VCからの資金調達可能性 |
| 差別化 | BYOキー＋マルチモデルは既存SaaSにない特徴 |

#### デメリット・リスク

| 項目 | 詳細 |
|------|------|
| 競合激化 | Cursor, GitHub Copilot Workspace, Devin, Replit Agent等が強力 |
| 開発リソース | 一人で商用品質を維持するのは困難 |
| 法的責任 | 利用規約、プライバシーポリシー、SLA対応 |
| インフラコスト | サーバー、DB、セキュリティ監査などのコスト |
| サポート負担 | 有料ユーザーへのサポート義務 |
| LLM API依存 | 各プロバイダーの利用規約・価格変更リスク |

## 競合分析

| 競合 | 価格帯 | 差別化ポイント |
|------|--------|----------------|
| Cursor | $20/月 | IDE統合、高い完成度 |

## 推奨: 段階的アプローチ

リスクを最小化しながら可能性を最大化するため、以下の段階的アプローチを推奨します。

```
Phase 1: OSS公開（0-6ヶ月）
    ↓
Phase 2: コミュニティ構築・フィードバック収集（6-12ヶ月）
    ↓
Phase 3: オープンコアへの移行検討（12ヶ月以降、成功した場合）
```

### なぜこのアプローチか

1. **リスク最小化**: 初期投資なしで市場検証できる
2. **柔軟性確保**: コミュニティの反応を見て方向転換可能
3. **差別化活用**: 「BYOキー + セルフホスト」は有料SaaSへのアンチテーゼとして訴求力がある
4. **現実的**: 一人での商用化は難しいが、OSSなら持続可能
